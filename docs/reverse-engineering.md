# Reverse Engineering Notes

## 当前已确认状态

FA860 的 Windows USB 控制路径已经确认可直接通过 HID 写入控制。当前已经明确并真机验证成功的 `A1` 控制家族包括：静音/取消静音、每通道信源使能控制、每通道音量控制，以及当前已收口的混音控制。

已确认设备信息：

- USB HID 设备：`VID_0483` / `PID_5750`
- Windows HID 接口路径可通过 `setupapi-hid` 获取
- 实际写入时需要关闭官方 `FA-860` 软件，否则 HID 接口会被独占
- HID report 长度为 64 字节，前 24 字节有效，后续补零

## 已验证 HID 路径

当前真机成功使用过的 HID 路径：

```text
\\?\hid#vid_0483&pid_5750#7&279e21ab&0&0000#{4d1e55b2-f16f-11cf-88cb-001111000030}
```

## 已验证控制帧结构

静音控制应使用 `A1` 帧族，结构如下：

```text
91 91 91 EE A1 01 00 04 CHANNEL_INDEX 1F 00 00 08 00 MUTE_FLAG 00 58 02 00 00 00 TARGET_VALUE DERIVED AA
```

字段含义：

- `CHANNEL_INDEX`：零基通道号
- `MUTE_FLAG=0x00`：静音
- `MUTE_FLAG=0x01`：取消静音
- `TARGET_VALUE`：通道目标值
- `DERIVED`：由 `TARGET_VALUE` 和通道号推导出的尾字节

当前信源控制帧可收敛成：

```text
91 91 91 EE A1 01 00 04 CHANNEL_INDEX 23 00 00 08 00 A4 01 14 00 14 00 01 SOURCE_MASK DERIVED AA
```

当前已经可以确认：

- `CHANNEL_INDEX` 在这类 `A1 23` 样本里确实会随通道变化：当前已看到 `CH1 -> 0x00`、`CH2 -> 0x01`、`CH3 -> 0x02`、`CH4 -> 0x03`
- `SOURCE_MASK` 仍然是三路输入源的位掩码
- 位定义保持不变：数字 = `0x01`，LINE = `0x02`，BLE = `0x04`
- `DERIVED` 不是常量异或，当前更像按通道变化：`DERIVED = SOURCE_MASK XOR (0x2B XOR CHANNEL_INDEX)`

已观察到的组合：

```text
0x07 -> 全开
0x05 -> 只关 LINE
0x03 -> 只关 BLE
0x06 -> 只关数字
```

用当前样本代入可验证：

- CH1 只关 LINE: `0x05 XOR (0x2B XOR 0x00) = 0x2E`
- CH1 全开: `0x07 XOR (0x2B XOR 0x00) = 0x2C`
- CH2 只关 LINE: `0x05 XOR (0x2B XOR 0x01) = 0x2F`
- CH2 全开: `0x07 XOR (0x2B XOR 0x01) = 0x2D`
- CH3 只关 LINE: `0x05 XOR (0x2B XOR 0x02) = 0x2C`
- CH3 全开: `0x07 XOR (0x2B XOR 0x02) = 0x2E`
- CH4 只关 LINE: `0x05 XOR (0x2B XOR 0x03) = 0x2D`
- CH4 全开: `0x07 XOR (0x2B XOR 0x03) = 0x2F`

这说明：

- `A1 23` 报文确实携带了按通道生效的信源控制信息；
- `CHANNEL_INDEX` 与 `SOURCE_MASK` 的组合已经可以直接用于生成工作中的信源控制命令；
- 旧的 `A2` 信源短帧更适合作为早期观察到的相关/同步帧保留，不再作为主控制模型。

## 已确认混音控制结构

从官方软件“输入信号”页面的 12x8 混音矩阵，以及后续补到的 CH1、CH2 样本来看，混音控制使用独立的 `A1 21` 帧族，并且 `BLOCK_INDEX = 0x01` 当前已经可以确认为 8 路 LINE 输入块，而不是之前暂定的 3 值块。

当前已拿到的样本：

```text
CH1 block1 = [100, 100, 100,   0,   0,   0,   0,   0] -> 91 91 91 EE A1 01 00 04 00 21 01 00 08 00 64 64 64 00 00 00 00 00 E8 AA
CH1 block1 = [100,   0, 100,   0,   0,   0,   0,   0] -> 91 91 91 EE A1 01 00 04 00 21 01 00 08 00 64 00 64 00 00 00 00 00 8C AA
CH1 block1 = [  0,   0, 100,   0,   0,   0,   0,   0] -> 91 91 91 EE A1 01 00 04 00 21 01 00 08 00 00 00 64 00 00 00 00 00 E8 AA
CH1 block1 = [  0,   0,   0,   0,   0,   0,   0,   0] -> 91 91 91 EE A1 01 00 04 00 21 01 00 08 00 00 00 00 00 00 00 00 00 8C AA
CH1 block1 = [  0,   0,   0, 100,   0,   0,   0,   0] -> 91 91 91 EE A1 01 00 04 00 21 01 00 08 00 00 00 00 64 00 00 00 00 E8 AA
CH2 block1 = [  0, 100,   0, 100,   0,   0,   0, 100] -> 91 91 91 EE A1 01 00 04 01 21 01 00 08 00 00 64 00 64 00 00 00 64 E9 AA
CH3 block1 = [  1,   2,   3,   4,   5,   6,   7,   8] -> 91 91 91 EE A1 01 00 04 02 21 01 00 08 00 01 02 03 04 05 06 07 08 86 AA
CH3 0x22 frame = [  0,   0, 100,   0,  11,  12,   9,  10] -> 91 91 91 EE A1 01 00 04 02 22 00 00 08 00 00 00 64 00 0B 0C 09 0A EC AA
```

结合截图、官方抓包和后续实机发送验证，当前可以确认：

- `CHANNEL_INDEX = 0x00` 对应 CH1
- `CHANNEL_INDEX = 0x01` 对应 CH2
- `CHANNEL_INDEX = 0x02` 对应 CH3
- `0x21` 是新的混音控制子类型
- `BLOCK_INDEX = 0x01` 对应每个输出通道的 LINE1 到 LINE8 输入块
- 第 15 到 22 字节是这一块 8 个输入的当前混音值
- 当前 `block 1` 已经可以解释为 `LINE1 / LINE2 / LINE3 / LINE4 / LINE5 / LINE6 / LINE7 / LINE8`
- `0x22` 是当前新观察到的第二类混音帧；它同样携带 8 个值，其中后四个值现在已经可以解释为数字左/右、蓝牙左/右

因此，这类样本当前可以收敛成：

```text
91 91 91 EE A1 01 00 04 CHANNEL_INDEX 21 BLOCK_INDEX 00 08 00 VALUE1 VALUE2 VALUE3 VALUE4 VALUE5 VALUE6 VALUE7 VALUE8 DERIVED AA
```

目前基于现有样本，可以确认：

- `block1` 至少覆盖 `LINE1..LINE8`
- `CH1 / block1` 的第 2 个值对应 `LINE2`
- `CH1 / block1` 的第 3 个值对应 `LINE3`
- `CH1 / block1` 的第 4 个值对应 `LINE4`
- `CH2 / block1` 的第 8 个值对应 `LINE8`
- `DERIVED` 满足 `VALUE1 XOR VALUE2 XOR VALUE3 XOR VALUE4 XOR VALUE5 XOR VALUE6 XOR VALUE7 XOR VALUE8 XOR seed`

对当前样本可得到：

- `CH1 / block1` 的 `seed = 0x8C`
- `CH2 / block1` 的 `seed = 0x8D`

这说明在当前已知范围内，seed 至少会随通道递增；当前代码先采用工作假设：

```text
seed = 0x8C + CHANNEL_INDEX + ((BLOCK_INDEX - 1) * 0x10)
```

- 当前项目中已经把 `A1 21 / index 1` 固化为 `LINE1..LINE8` 的正式工作模型
- 当前项目中已经把 `A1 22 / index 0` 固化为 `DIGITAL_L`、`DIGITAL_R`、`BT_L`、`BT_R` 的正式工作模型

## 已确认 0x22 混音帧结构

目前已经抓到第一条 `A1 22` 样本：

```text
CH3 -> 91 91 91 EE A1 01 00 04 02 22 00 00 08 00 00 00 64 00 0B 0C 09 0A EC AA
```

当前可以确认这些：

- `0x22` 与 `0x21` 同属 `A1` 混音控制家族
- 它同样携带 8 个数据字节
- 这条 CH3 样本的原始值区是 `[0, 0, 100, 0, 11, 12, 9, 10]`
- 如果把整块 8 个值一起参与异或，则当前样本的 seed 候选值为 `0x8C`
- 后四个值 `11, 12, 9, 10` 对应 12 路混音里的最后四路：`DIGITAL_L`、`DIGITAL_R`、`BT_L`、`BT_R`

因此，当前 12 路混音的工作模型可以先写成：

- `0x21 / index 1` -> `LINE1..LINE8`
- `0x22 / index 0` 的后四位 -> `DIGITAL_L`、`DIGITAL_R`、`BT_L`、`BT_R`
- `0x22 / index 0` 的前四位 -> 当前看起来是“每通道各自固定的模板前缀”，不是全通道共用同一组字节

当前已经确认：

- 这条 `0x22` 帧承载的是 `LINE1..LINE8` 之外的混音尾段
- 样本尾 4 个值已经和界面设置完成一一对应：`DIGITAL_L`、`DIGITAL_R`、`BT_L`、`BT_R`
- `0x22` 前四位当前按通道奇偶交替前缀处理
- `0x22` 的 seed 当前按 `0x8E XOR CHANNEL_INDEX` 处理
- 该模型已经过 CH1-CH8 官方样本和后续实机发送验证共同支持

当前已观察到的通道前缀/seed：


这组 CH1-CH8 样本已经足够先收敛出两个工作规律：

```text
	00 00 64 00  if CHANNEL_INDEX is even
```

## 音量控制结构

目前已经拿到连续样本、跨通道样本和外推实机命中样本，已经足够形成当前的音量工作模型。

当前样本：

```text
CH1 -9 dB  -> 91 91 91 EE A1 01 00 04 00 1F 00 00 08 00 01 00 FE 01 00 00 00 01 4C AA
CH1 -10 dB -> 91 91 91 EE A1 01 00 04 00 1F 00 00 08 00 01 00 F4 01 00 00 00 01 46 AA
CH1 -10 dB 静音   -> 91 91 91 EE A1 01 00 04 00 1F 00 00 08 00 00 00 F4 01 00 00 00 01 47 AA
CH1 -10 dB 取消静音 -> 91 91 91 EE A1 01 00 04 00 1F 00 00 08 00 01 00 F4 01 00 00 00 01 46 AA
CH2 -15 dB -> 91 91 91 EE A1 01 00 04 01 1F 00 00 08 00 01 00 C2 01 00 00 00 05 75 AA
CH7 -20 dB -> 91 91 91 EE A1 01 00 04 06 1F 00 00 08 00 01 00 90 01 00 00 00 15 30 AA
```

从字面上看，它与静音帧仍然共享前半段：

```text
91 91 91 EE A1 01 00 04 CHANNEL_INDEX 1F 00 00 08 00 VALUE_LO VALUE_HI PARAM_LO PARAM_HI 00 00 00 TARGET DERIVED AA
```

当前可以确认这些观察：

- `CHANNEL_INDEX=0x00` 对应 CH1
- 这一类帧仍然落在 `A1` / `1F` 这条控制族上
- 第 17-18 字节看起来不是固定参数标识，而是音量值本身的小端编码
- `TARGET` 仍然沿用每个通道已有的对象值：CH1 = `0x01`，CH2 = `0x05`

基于 `-9 dB` 与 `-10 dB` 这两个连续样本，当前最强的候选规律是：

- 音量值很可能编码在第 17-18 字节（小端序）
- `-9 dB -> 0x01FE = 510`
- `-10 dB -> 0x01F4 = 500`
- 每降低 `1 dB`，编码值减少 `10`
- CH2 `-15 dB -> 0x01C2 = 450`，也与同一条线性关系完全一致

如果这个线性关系成立，那么编码很可能是：

```text
encoded_volume = (db + 60) * 10
```

其中 `db` 取值范围是 `-60 .. 0`。这样会得到：

- `0 dB   -> 600  -> 0x0258`
- `-9 dB  -> 510  -> 0x01FE`
- `-10 dB -> 500  -> 0x01F4`
- `-15 dB -> 450  -> 0x01C2`
- `-20 dB -> 400  -> 0x0190`
- `-60 dB -> 0    -> 0x0000`

这和用户给出的“0 dB 到 -60 dB（关闭）”范围正好一致。

另外，`-10 dB` 的静音/取消静音成对样本表明，这条音量帧里也复用了和静音帧一致的 `FLAG` 位：

- `FLAG=0x00` -> 当前音量下的静音状态
- `FLAG=0x01` -> 当前音量下的非静音状态

对目前样本，尾字节还可以整理成下面这条候选规律：

```text
DERIVED = value_lo XOR value_hi XOR TARGET_VALUE XOR CHANNEL_INDEX XOR seed
```

其中：

- mute -> `seed = 0xB3`
- unmute -> `seed = 0xB2`

现有样本都满足：

- CH1 -10 mute : `F4 ^ 01 ^ 01 ^ 00 ^ B3 = 47`
- CH1 -10 unmute : `F4 ^ 01 ^ 01 ^ 00 ^ B2 = 46`
- CH1 -9 unmute : `FE ^ 01 ^ 01 ^ 00 ^ B2 = 4C`
- CH2 -15 unmute : `C2 ^ 01 ^ 05 ^ 01 ^ B2 = 75`
- CH7 -20 unmute : `90 ^ 01 ^ 15 ^ 06 ^ B2 = 30`

需要特别注意：你标成“CH1 恢复 0” 的第二条样本：

```text
91 91 91 EE A1 01 00 04 00 1F 00 00 08 00 01 00 58 02 00 00 00 01 E9 AA
```

这条虽然非常像 `0 dB -> 0x0258` 的线性编码结果，但它同时也与之前已经确认过的 CH1 取消静音帧完全一致：

- 参数标识仍然是 `58 02`
- 尾字节 `E9` 也完全匹配 CH1 unmute

因此，当前真实状态应表述为：

- 音量已经拿到两条连续样本：`CH1 -9 dB`、`CH1 -10 dB`
- 还拿到了 `CH1 -10 dB` 的静音/取消静音成对样本，以及 `CH2 -15 dB` 的跨通道样本
- `CH7 -20 dB` 已经通过真机直接写入验证成功
- `CH7 -30 dB`、`CH2 -20 dB` 也已经通过真机直接写入验证成功
- `0 dB` 这一点的编码从线性关系上看很像 `0x0258`
- 但 `0x0258` 这条帧目前与“CH1 取消静音”字节完全重合，语义仍有歧义
- 现有数据已经足够支持当前代码中的正式音量控制入口
